## 🏗️ 아키텍처 — 놀랍도록 단순한 구조

이온로드는 **의존성 하나(better-sqlite3)**로 돌아가는 풀스택 앱입니다.

```
┌─────────────┐     HTTPS      ┌──────────────┐     SQLite      ┌──────────┐
│   📱 모바일   │ ──────────── │  Node.js 서버  │ ──────────── │  /data/   │
│  (바닐라 JS)  │              │  (순수 http)   │              │  survey.db│
└─────────────┘               └──────────────┘               └──────────┘
       │                            │
       │                     ┌──────┴──────┐
       │                     │ 카카오맵 API  │
       │                     │ (주소/지도)   │
       └────────────────────└─────────────┘
```

**왜 이렇게 단순한가?**

React? Vue? Express? PostgreSQL? 전부 고려했지만 다 빼버렸습니다. 이유:

1. **사용자가 20명**: 동시 접속 ~20명이면 SQLite WAL 모드로 충분합니다. PostgreSQL은 총 들고 모기 잡는 격.
2. **앱 설치 불가**: 현장 직원들에게 "npm install 하세요"는 불가능. **링크 하나**로 접속 가능해야 합니다.
3. **배포 단순성**: Railway에 Dockerfile 하나 올리면 끝. CI/CD 파이프라인? 없어도 됩니다.

이건 "스타트업 MVP" 철학과 같습니다. **작동하는 최소한의 것**을 빠르게 만들고, 필요할 때 확장하는 것.

## 📁 코드베이스 지도

```
src/
├── server.js          ← 웹서버 (순수 Node.js http, 라우팅, 인증, 정적 파일 서빙)
├── config.js          ← 환경변수 + 기본 설정
├── catalog.js         ← 제품/지역/매장유형 목록 (하드코딩, 추후 DB화 가능)
├── geocoding.js       ← 카카오 REST API 주소→좌표 변환 + 캐시
├── assignment.js      ← 거리+공정성 기반 조사 지역 자동 배정
├── storage/
│   ├── index.js       ← 스토리지 추상화 (로컬 + 구글시트 미러)
│   ├── sqliteStore.js ← SQLite CRUD (WAL 모드, 지오캐시)
│   └── ...기타        ← Google Sheets 연동 (미사용)
└── client/
    ├── index.html     ← 메인 앱 셸 (PWA, 스켈레톤 로딩)
    ├── app.js         ← 클라이언트 로직 (폼 위저드, 대시보드, 지도)
    ├── admin.html     ← 관리자 페이지
    ├── admin.js       ← 관리자 로직 (통계, CSV, 필터)
    ├── styles.css     ← 디자인 시스템 (다크모드 포함)
    └── manifest.json  ← PWA 매니페스트
```

### 연결 관계 — 데이터가 흐르는 길

1. **사용자가 기록 제출** → `app.js` → `POST /api/submissions` → `server.js`
2. **서버가 검증** → `validateSubmission()` (길이, 범위, 사진 크기)
3. **지오코딩** → `geocoding.js` → 카카오 API로 주소→좌표 변환
4. **배정 로직** → `assignment.js` → haversine 거리 + 공정성 가중치
5. **DB 저장** → `sqliteStore.js` → `/data/survey.db`
6. **대시보드 갱신** → `GET /api/bootstrap` → 클라이언트 30초 폴링

## 🎯 기술적 결정들 — "왜 그렇게 했는가"

### SQLite를 선택한 이유 (PostgreSQL 대신)
비유하면, 마을 구멍가게에 대형마트 물류시스템을 설치하는 격입니다. 20명이 쓰는 앱에 연결 풀링, 트랜잭션 격리 레벨, 레플리카... 필요 없습니다. SQLite는 단일 파일이라 백업도 `cp` 한 번이면 끝.

### 카카오맵 (Google Maps 대신)
한국 주소체계에는 카카오가 압도적입니다. "GS25 역삼점"을 검색하면 정확한 주소+좌표가 나옵니다. Google Maps에선 한국 상호명 검색이 불안정하고, 무료 할당량도 카카오(30만/일)가 훨씬 넉넉합니다.

### 바닐라 JS (React 대신)
- 번들 사이즈: 0KB (빌드 단계 없음)
- 학습 곡선: HTML 아는 사람이면 수정 가능
- 트레이드오프: 코드가 좀 더 장황하지만, 1,400줄이면 관리 가능한 수준

### `autoload=false` + `kakao.maps.load()`
이건 **버그와의 전쟁**에서 배운 교훈입니다. 카카오 SDK를 `<script>`로 로드하면 `kakao.maps`는 존재하지만 `kakao.maps.LatLng`는 아직 없는 타이밍 이슈가 생깁니다. `autoload=false`로 두고 명시적으로 `kakao.maps.load(callback)`을 호출하면 확실히 초기화된 후에 사용할 수 있습니다.

## 🐛 버그 전쟁 — 우리가 밟은 지뢰들

### 1. "Missing required submission fields" 미스터리
**증상**: 폼 입력 후 제출하면 400 에러.
**원인**: 3단계 폼 위저드에서 Step 1 데이터를 localStorage에 저장하는 타이밍 문제. "다음" 버튼 클릭 이벤트가 두 곳에 있었고, 하나만 localStorage에 저장했습니다.
**해결**: state 객체 + localStorage 이중 백업. 제출 시 둘 다에서 읽어서 null 방지.
**교훈**: 멀티스텝 폼에서는 각 스텝의 데이터를 **즉시, 확실하게** 저장해야 합니다.

### 2. 카카오맵 "domain mismatched!" 에러
**증상**: 맵이 안 뜨고 콘솔에 domain mismatched 에러.
**원인**: 카카오 개발자 콘솔에 `http://` 도메인을 등록했는데 Railway는 `https://`를 사용.
**해결**: 정확한 HTTPS 도메인 등록.
**교훈**: API 키 설정 시 프로토콜까지 정확히 맞춰야 합니다.

### 3. 지도 마커가 안 뜨는 문제
**증상**: 데이터는 있는데 마커가 안 보임.
**원인**: 좌표가 `submission.survey.coordinates.lat`에 있는데 코드는 `submission.lat`을 찾고 있었음. API 응답 구조와 클라이언트 코드의 불일치.
**해결**: 여러 경로에서 좌표를 탐색하는 폴백 체인.
**교훈**: API 응답 구조가 바뀌면 프론트엔드도 반드시 함께 업데이트.

### 4. 백업 스크립트 401 Unauthorized
**증상**: 매일 밤 백업 크론잡이 실패.
**원인**: 어드민 인증을 추가하면서 `/api/backup` 엔드포인트도 보호됐는데, 백업 스크립트는 인증 없이 호출.
**해결**: 스크립트에 로그인→토큰 획득→인증 헤더 추가.
**교훈**: 보안 기능 추가 시 **자동화 스크립트도 함께** 업데이트해야 합니다.

## 🏅 좋은 엔지니어가 하는 것들 — 이 프로젝트에서 배울 점

### 1. "작동하는 것"을 먼저, "완벽한 것"은 나중에
첫 커밋은 JSON 파일 저장이었습니다. SQLite? 나중에. 인증? 나중에. 먼저 **사용 가능한 상태**를 만들고, 점진적으로 강화했습니다.

### 2. 테스트는 안전망
17개 테스트가 매 커밋마다 돌아갑니다. 지역명을 영어→한글로 바꿀 때 7개 파일을 수정했지만, 테스트가 있어서 자신있게 변경할 수 있었습니다.

### 3. 사용자 데이터로 판단하라
Round 4에서 실제 제출 데이터를 보니 가격 0건인 제출이 많았습니다. 이걸 보고 "가격 미입력 리마인더"를 추가했습니다. **추측이 아닌 데이터로 결정**하는 것.

### 4. 보안은 처음부터 (하지만 과도하게는 말고)
Rate limiting, 입력 검증, CORS — 작은 앱이라도 기본 보안은 갖춰야 합니다. 하지만 JWT, OAuth, RBAC까지 갈 필요는 없었습니다. 비밀번호 하나면 충분한 규모.

### 5. 오프라인을 고려하라
현장 조사는 지하, 산간지역에서도 합니다. 오프라인 제출 큐를 넣어서 연결 끊겨도 데이터가 유실되지 않게 했습니다.
